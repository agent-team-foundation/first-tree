# Proposal: Governance Domain

**Author:** yuezengwu
**Date:** 2026-03-24
**Status:** Draft

---

## Summary

Add a `governance/` domain to Context Tree, providing full-lifecycle quality, compliance, and security through a four-layer architecture.

---

## Problem

The tree currently has structural linting (validate_nodes.py) and AI PR review as quality gates, but lacks:
- Drift detection between merges (nodes going stale without anyone noticing)
- Semantic auditing (cross-node contradictions, redundant work, direction misalignment)
- A unified framework that explains how these layers relate and when each applies

---

## Proposed Architecture

Four layers, from cheapest to deepest:

### Layer 1: Standards — Define what is correct

Declarative rules that already exist as documents. Zero runtime cost, always in effect. This layer requires no new implementation — the governance NODE.md simply references existing standard files:

- `principles.md` — Core principles: decisions not execution, agents as first-class participants, transparency, git-native structure
- `ownership-and-naming.md` — Naming conventions, ownership model (explicit owners, `[]` inheritance, `[*]` wildcard)
- `AGENT.md` — Agent workflow: read before acting, decide in the tree, update after every task

### Layer 2: Gates — Block bad changes before merge

Synchronous checks on every PR. Fast, deterministic, blocking. Existing implementation enhanced by this proposal:

| Gate | Enforces | Mechanism | Status |
| --- | --- | --- | --- |
| Structural lint | Frontmatter validity, owners format, soft_link resolution, folder structure | `.scripts/validate_nodes.py` in PR CI | Running |
| Structural lint (new) | Directory-listing consistency, root-domain sync, soft-link reciprocity, empty nodes, title mismatch | Same as above (5 new checks in this PR) | This PR |
| AI PR review | Principles compliance, consistency with existing nodes, soft_link usage, clarity, ownership respect | `.scripts/run_review.py` + `.github/prompts/pr-review.md` | Running |
| CODEOWNERS | Owner approval required for changes to their nodes | `.github/CODEOWNERS` (auto-generated by `generate_codeowners.py`) | Running |

### Layer 3: Monitors — Detect drift between merges

Asynchronous periodic checks using git history. No AI, no external data. Monitors do not claim nodes are wrong — they rank nodes by drift probability so someone can go check.

Requires new `.scripts/tree_freshness.py` implementing 4 detection types:

| Monitor | Logic | Signal |
| --- | --- | --- |
| Stale node | `git log` to find leaf nodes not updated in >90 days | Low (old ≠ wrong) |
| Stale domain | Entire domain directory with no file changes in >90 days | Medium (forgotten domain) |
| Proposal lifecycle | Files in `proposals/` older than 30 days still unresolved (calculated from filename date) | Medium (stuck decision) |
| Hot-domain-cold-node | Node >90 days stale inside an actively evolving domain | High (surrounding context has moved on) |

Deduplication rules:
- Nodes flagged by hot-domain-cold-node are excluded from stale-node results (the former is strictly more informative)
- Child domains are excluded from stale-domain if a parent domain is already flagged

Execution: new `.github/workflows/tree-freshness.yml`, triggered weekly on Monday 09:00 UTC. Output auto-creates GitHub Issues assigned to the corresponding node owners. Findings unresolved after 14 days escalate to root tree owners.

### Layer 4: Audits — Find what no other layer can

Periodic AI-powered semantic analysis. Catches problems invisible to deterministic checks.

Requires new `.scripts/tree_audit.py` with a 6-step workflow:

1. **Prepare** (deterministic) — Parse all nodes, build soft_link graph, collect proposals and recent git changes
2. **Conflicts** (AI × N calls) — Pairwise conflict check along the soft_link graph, one AI call per related node pair
3. **Redundancy** (AI × 1 call) — Check proposals + recent changes for duplicated effort or hidden dependencies
4. **Direction** (AI × 1 call) — Check whether recent activity diverges from declared strategic direction
5. **Verify** (AI × 1 call) — Adversarial verification: review each candidate finding, require direct quotes from source nodes, filter out unsupported findings
6. **Report** (deterministic) — Aggregate results, resolve owners, produce output

Requires 4 new AI prompts in `.github/prompts/audit/`:

| Prompt | Input | Output |
| --- | --- | --- |
| `audit-conflicts.md` | Full text of two soft_link-connected nodes | JSON array: decision_conflict / constraint_violation / terminology_inconsistency, must quote both sides |
| `audit-redundancy.md` | All active proposals + nodes changed in last 30 days | JSON array: duplicated_effort / hidden_dependency / parallel_collision, must cite evidence |
| `audit-direction.md` | Strategic docs (root NODE.md, infrastructure.md, domain NODE.md files) + last 30 days git history | JSON array: stalled_direction / undeclared_direction / drift |
| `audit-verify.md` | Candidate findings + full text of referenced nodes | JSON: verified[] + rejected[], confirming or rejecting each finding |

Trigger: monthly / on-demand. Budget cap: $5 per run.

### How layers work together

Standards define correctness -> Gates enforce at change time -> Monitors detect accumulated drift -> Audits find hidden contradictions.

Each layer catches what the layer above cannot. A finding at any layer can feed back to update Standards or tighten Gates.

---

## Key Decisions

- **Report, don't fix.** Findings are reported as GitHub issues. Auto-fix risks resolving conflicts the wrong way.
- **Pre-merge is the only synchronous gate.** Monitors and Audits are advisory — keeps the developer loop fast.
- **Pairwise comparison for conflict detection.** Follow soft_link graph, compare node pairs individually. Small context per call enables deeper reasoning.
- **Adversarial verification for AI findings.** All audit findings must be justified with direct quotes.
- **Self-governance first, external service later.** This tree is the first use case before offering governance as a service.

---

## Governance Boundaries

- Does not validate source systems against the tree (that's each domain's responsibility).
- Does not auto-fix (humans/agents decide resolution).
